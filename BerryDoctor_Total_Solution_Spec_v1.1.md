## 1. Executive Summary

BerryDoctor는 영세 딸기 농가를 위한 **무료 오픈소스 토털 스마트팜 솔루션**이다.
일본(소농 표준 패키지)·중국(AI 196% 증산)·한국(ICT 인프라) 3국의 장점을 결합하여,
**하드웨어 30~50만원 + 소프트웨어 0원**으로 프리바(Priva)급 환경제어를 실현한다.

### 1.1 핵심 수치 목표

| 지표 | 전통 농가 | BerryDoctor 적용 후 |
|------|-----------|-------------------|
| 하우스 방문 횟수/일 | 3~5회 | 0~1회 (긴급 시만) |
| 양액 EC 오차 | ±0.5 mS/cm | ±0.1 mS/cm |
| 병해 발견 시점 | 육안 (늦음) | AI 조기 감지 (1~3일 빠름) |
| 소프트웨어 비용 | 연 100~300만원 | 0원 |
| 하드웨어 총비용 | 1,500~3,000만원/동 | 30~50만원/동 |

## 2. 3국 기술 비교 분석 및 채택 전략

### 2.1 일본 — 소농 표준 패키지 (채택: ★★★)

**장점:**
- UECS(Ubiquitous Environment Control System) 플랫폼: Arduino 기반, 약 6만엔(~60만원)
- 농민 정보 리터러시에 의존하지 않는 "기본 설정만으로 효율 향상" 철학
- 소규모 온실(300~1,000㎡)에 최적화된 설계
- 2년간 실증 검증 완료 (딸기 수경재배)

**단점:**
- UI가 엔지니어 중심 (농부 친화적이지 않음)
- 클라우드 연동 미흡, 원격 제어 제한적
- AI/ML 의사결정 기능 없음

**채택 요소:**
→ 하드웨어 구성 (ESP32 + 저가 센서 조합)
→ PID 기반 양액 제어 알고리즘
→ "설정 안 해도 작동하는" 기본값 전략

### 2.2 중국 — AI 의사결정 엔진 (채택: ★★★)

**장점:**
- CyberFarmer.HortiGraph: Knowledge Graph + 이미지인식 + 기후모델 통합
- 딸기 생산량 196% 증가, ROI 75.5% 향상 (FAO 검증)
- 원격 제어 실증: 팀이 물리적으로 온실에 가지 않고 4개월간 원격 재배
- Zhi Duo Mei 팀의 상용화 → 실제 농협에 배포 검증

**단점:**
- 고급 센서 + 유리온실 전제 (소농에 비현실적)
- 클라우드 의존 (중국 서버)
- 오픈소스가 아님

**채택 요소:**
→ Knowledge Graph 구조 (농촌진흥청 데이터 기반으로 재구축)
→ 이미지 기반 생육 판단 파이프라인
→ 양액·기후 통합 의사결정 알고리즘
→ "농부가 안 가도 되는" 원격 운영 철학

### 2.3 한국 — ICT 인프라 + 정책 연계 (채택: ★★☆)

**장점:**
- 스마트팜코리아 공공 데이터셋 (환경·제어·생육·경영·이미지)
- 농사로 매뉴얼 + 농촌진흥청 기술 지원 체계
- 카카오톡 보급률 95% → 별도 앱 설치 없이 접근 가능
- 1세대 스마트팜 보급 경험 축적

**단점:**
- 소프트웨어가 대부분 폐쇄적 상용 제품
- 농가 자부담 수천만원 구조
- UI/UX가 엔지니어 중심, 고령 농부 배려 부족

**채택 요소:**
→ 카카오톡 챗봇 인터페이스 (한국 농부 최적 접점)
→ 공공 데이터 API (기상청, 도매시장 경락가, 농사로)
→ 정부 보조금 매칭 기능

### 2.4 통합 채택 매트릭스

| 기능 영역 | 일본 방식 | 중국 방식 | 한국 방식 | BerryDoctor |
|-----------|----------|----------|----------|-------------|
| 하드웨어 | Arduino+UECS | 고급 센서+유리온실 | 상용 ICT패키지 | **ESP32+저가센서 (일본식)** |
| 제어 알고리즘 | PID 단순 제어 | AI+Knowledge Graph | 복합환경제어기 | **PID+룰엔진+로컬LLM (일+중)** |
| 양액 관리 | EC/pH 모니터링 | AI 자동 배합 | Nutriflex 등 유료 | **PID 자동배합 (일+중)** |
| 병해 감지 | 수동 | 이미지 인식 AI | 컨설턴트 의존 | **YOLOv5 스마트폰 (중국식)** |
| UI/접점 | 웹 대시보드 | 클라우드 앱 | 전용 SW | **카카오톡 봇 (한국식)** |
| 데이터 | 로컬 CSV | 클라우드 빅데이터 | 스마트팜코리아 | **로컬+선택적 공유 (전체)** |
| 비용 | ~60만원 | 수천만원 | 1,500만~3,000만원 | **30~50만원** |

| **Tier 1** | 기본 모니터링 | **~5만원** | 온습도/조도/배지온도 실시간 모니터링 |

| **참고: 프리바** | 풀 시스템 | **3,000만원~** | 동일 기능 |

## 5. 소프트웨어 아키텍처

### 5.1 기술 스택

| 계층 | 기술 | 역할 |
|------|------|------|
| 코어 엔진 | **berry-engine** (Python 3.11) | 센서수집→판단→제어→알림 전체 파이프라인, 단일 프로세스 |
| 데이터베이스 | **SQLite3** (단일 .db 파일) | 센서 로그, 재배 일지, 설정값. 설치 불필요 |
| MQTT 브로커 | **Mosquitto** (내장, 자동 실행) | ESP32 ↔ berry-engine 통신 |
| 스케줄러 | **APScheduler** (Python 내장) | 정기 작업: 기상 확인, 시세 조회, 촬영 트리거 |
| 병해 AI | **YOLOv5s** (PyTorch/ONNX) | 딸기 7대 질병 이미지 분류 |
| 로컬 LLM | **Qwen2.5-7B** (QLoRA, 선택사항) | 재배 Q&A, 의사결정 보조. GPU 있는 PC만 |
| 시세 예측 | **Prophet** / **LightGBM** | 도매시장 경락가 시계열 예측 |
| 챗봇 | **카카오톡 채널 API** | 농부 유일한 인터페이스. 알림+명령 |
| 데스크톱 UI | **시스템 트레이** (pystray) | 초록불=정상, 빨간불=문제. 클릭→웹 대시보드 |
| 웹 대시보드 | **FastAPI + Jinja2** (localhost:8080) | 차트, 이력, 설정. 선택적 사용 |
| MCU 펌웨어 | **Arduino Framework** (PlatformIO) | ESP32 센서+릴레이 제어 |
| 배포 | **PyInstaller** → 단일 .exe | 더블클릭 설치, 가입 없음, 의존성 없음 |

> **외부 가입이 필요한 서비스: 0개**
> 카카오톡 채널은 BerryDoctor Project가 운영 → 농부는 "친구 추가"만 하면 됨

### 5.2 berry-engine 내부 구조

```python

│
├── rules/
│   ├── engine.py            # 룰 엔진 (조건→액션 매핑)

│   ├── nutrient.py          # EC/pH PID 제어기

├── scheduler/
│   ├── jobs.py              # APScheduler 작업 등록
│   ├── weather.py           # 기상청 API 호출 (매 시간)
│   ├── market.py            # 도매시장 경락가 (매일 06:00)
│   └── camera.py            # ESP32-CAM 촬영 트리거 (매일 10:00)
│
├── ai/
│   ├── disease.py           # YOLOv5 병해 진단
│   ├── knowledge.py         # 품종·생육단계 Knowledge Graph

│
├── kakao/
│   ├── webhook.py           # 카카오톡 수신 (Flask 미니 서버)
│   ├── sender.py            # 카카오톡 알림 발송
│   └── commands.py          # 명령어 파싱 ("상태", "환풍기 켜" 등)
│

│
├── web/

│

    └── icon.py              # 시스템 트레이 아이콘 (pystray)
```

### 5.3 berry-engine 실행 흐름

```
[농부: 딸기박사.exe 더블클릭]
    │
    ├─→ Mosquitto 자동 시작 (localhost:1883)
    ├─→ SQLite DB 초기화 (berry.db 없으면 생성)
    ├─→ 시스템 트레이 아이콘 표시 (🟢 정상)
    ├─→ FastAPI 웹서버 시작 (localhost:8080)
    ├─→ 카카오톡 웹훅 리스너 시작
    ├─→ MQTT 구독 시작 (sensor/#)
    └─→ APScheduler 작업 등록
         ├─ 매 5초: 센서 데이터 처리 + 룰 엔진 판단
         ├─ 매 시간: 기상청 API → 동해/고온 예측
         ├─ 매일 06:00: 도매시장 시세 → 출하 추천
         ├─ 매일 10:00: ESP32-CAM → YOLO 진단
         └─ 매일 21:00: 일일 리포트 카카오톡 발송

[ESP32 센서 데이터 수신 시]
    │
    ├─→ SQLite 저장
    ├─→ 룰 엔진 평가:
    │    if 온도 > 30°C → MQTT publish: relay/fan/on
    │    if 습도 > 90%  → MQTT publish: relay/vent/on
    │    if EC < 설정값  → MQTT publish: pump/nutrient/on
    │    if 야간 + 외기온 < 5°C → MQTT publish: relay/curtain/close
    └─→ 이상 시 → 카카오톡 즉시 알림

[농부 카카오톡 명령 수신 시]
    │
    ├─ "상태" → DB 최신값 조회 → 카카오톡 응답
    ├─ "환풍기 켜" → MQTT publish: relay/fan/on → 확인 응답
    ├─ "목표온도 22도" → config 업데이트 → 확인 응답
    ├─ "진단" + 사진 → YOLO 추론 → 결과 응답
    └─ "오늘 할일" → Knowledge Graph + 기상 + 생육단계 → 작업 목록
```

| GPIO13 | OUT | 릴레이 CH2 (커튼모터) | Digital | LOW=ON |
| GPIO14 | OUT | 릴레이 CH3 (CO2) | Digital | LOW=ON |
| GPIO27 | OUT | 릴레이 CH4 (예비) | Digital | LOW=ON |
| GPIO25 | OUT | MOSFET→밸브 #1 | PWM/Digital | |

| GPIO35 | ADC IN | SEN0161 pH 센서 | Analog | ADC1 only |
| GPIO36 | ADC IN | (예비 아날로그) | Analog | |

## 8. 배포 방식

### 8.1 농부용 (기술 지식 없음) — 99%의 사용자

```
1. 딸기박사_설치.exe 더블클릭
   → Python, Mosquitto, AI 모델 전부 자동 설치
   → 바탕화면에 "딸기박사" 아이콘 생성
   → 시스템 트레이에 🍓 아이콘 상주

2. 처음 실행 시 물어보는 것 (딱 3가지):
   → "WiFi 이름과 비밀번호"
   → "재배 품종 선택" (설향/금실/매향/기타)
   → "카카오톡에서 딸기박사 채널 추가"

3. 끝. 이후 모든 조작은 카카오톡으로.
```

> **가입 필요: 없음 | 로그인 필요: 없음 | 브라우저 열 필요: 없음**
> 컴퓨터 켜져 있으면 자동 실행, 꺼지면 ESP32가 독립 동작 (로컬 룰)

### 8.2 기술 농부 / 청년 농부용

1. GitHub에서 전체 소스 클론
2. `pip install -r requirements.txt`
3. `python main.py` → Mosquitto + berry-engine + 웹 대시보드 자동 구동
4. ESP32 펌웨어: PlatformIO로 빌드·업로드
5. 웹 대시보드: `http://localhost:8080` 에서 상세 차트·설정 가능

### 8.3 지자체 / 농업기술센터용

1. 센터 PC에 딸기박사.exe 설치 (관내 농가 통합 관리 모드)
2. 관내 농가 ESP32 등록 (각 농가별 MQTT 토픽 분리)
3. 카카오톡 일괄 등록
4. 웹 대시보드에서 관내 전체 농가 현황 모니터링

### 10.2 대체 부품 호환표

| 정규 부품 | 대체 가능 | 가격차 | 주의사항 |

## 11. 안전 주의사항
